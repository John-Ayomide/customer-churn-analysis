"Initial project setup — folder structure and dependencies"

Commands run and what they do

Step 1 — Create folder structure
mkdir data mkdir notebooks mkdir src mkdir reports
What this does: Creates four empty folders inside your project. Each has a specific role. data/ stores all CSV files. notebooks/ holds Jupyter analysis files. src/ is for reusable Python scripts later. reports/ is where charts and PDF outputs are saved.

Why run them separately on Windows: Windows PowerShell mkdir can only create one folder at a time, unlike Linux/Mac which accepts multiple names in one command.
Step 2 — Virtual environment
python -m venv venv venv\Scripts\activate
What this does: Creates an isolated Python environment just for this project. When activated, any library you install only lives inside this project folder — it will not affect other Python projects on your computer.

Why this matters: Without a virtual environment, all projects share the same libraries. Project A needs Pandas 1.5, Project B needs Pandas 2.0 — they break each other. venv solves this completely.

How you know it worked: You see (venv) at the start of your terminal line.
Step 3 — Install libraries and save dependencies
pip install pandas numpy matplotlib seaborn jupyter pip freeze > requirements.txt
What this does: Installs five core data science libraries. pip freeze > requirements.txt saves the exact version of every installed library into a text file.

Why requirements.txt matters: Anyone who clones your GitHub repo can run pip install -r requirements.txt and get exactly the same libraries. This makes your project reproducible — a professional standard recruiters notice.
Step 4 — GitHub setup
git init git add . git commit -m "Initial project setup..." git remote add origin https://github.com/... git push -u origin master
What each line does:

git init — turns this folder into a Git repository and starts tracking changes
git add . — stages all files, telling Git to include everything in the next save
git commit -m — saves a snapshot of your project with a message
git remote add origin — connects your local project to your GitHub repository online
git push — uploads your code to GitHub so it is live and visible
Purpose of Day 1
Day 1 is never about analysis — it is about building a professional foundation. Every real-world data project starts with structure. Without it, projects become messy fast: files everywhere, no version history, libraries conflicting. One hour here means every future hour is spent on actual analysis, not firefighting.
Goal achieved
A clean professional project skeleton is live on GitHub with a real commit history from Day 1. Recruiters who visit your GitHub see organised structure immediately — before they read a single line of your analysis code.
Alternative approaches
You could use Conda instead of venv — it manages both Python versions and libraries together. More powerful but more complex for a beginner. Poetry is becoming the modern standard for dependency management and is worth learning after you are comfortable with venv.

Every code cell explained

Cell 1 — Import libraries
import pandas as pd import numpy as np
What this does: Loads two libraries into memory so we can use their functions throughout the notebook.

pandas is the core tool for working with data tables in Python — a programmable Excel. as pd is a short nickname used universally by data analysts worldwide.

numpy provides fast mathematical operations. Pandas uses it internally. We import it as np by convention.
Cell 2 — Load the data
df = pd.read_csv('../data/WA_Fn-UseC_-Telco-Customer-Churn.csv') print("Shape:", df.shape)
What this does: Reads the CSV file and stores it in df — short for DataFrame, the standard name used by every pandas analyst.

df.shape returns (rows, columns). We saw (7043, 21) — 7,043 customers with 21 pieces of information each.

The ../data/ path: The .. means go up one folder. From notebooks/ we go up to the project root, then into data/. This keeps paths relative so the project works on any computer.
Cell 3 — Check data types
df.dtypes
What this does: Shows the data type of every column.

The three types we care about:
object — text strings like "Yes" or "Month-to-month"
int64 — whole numbers like 12 months of tenure
float64 — decimal numbers like 29.85 monthly charges

What we flagged: TotalCharges showed as object when it should be a number. This data quality issue was fixed on Day 3.
Cell 4 — Check missing values
df.isnull().sum()
What this does: Counts empty cells in every column. isnull() returns True/False for each cell. .sum() adds up all the True values since True equals 1 in Python.

Why always check this early: Missing values silently skew averages, break ML models, and cause errors. Finding them on Day 2 means you can plan how to fix them properly before analysis begins.
Cell 5 — Calculate the churn rate
print(df['Churn'].value_counts()) print(df['Churn'].value_counts(normalize=True).round(3))
What this does: value_counts() counts how many times each unique value appears. normalize=True converts counts to proportions. .round(3) rounds to 3 decimal places.

Result: No: 5,174 (73.5%) | Yes: 1,869 (26.5%)

Why 26.5% is significant: The telecom industry average is 10-15%. At 26.5% this company loses customers at nearly double the normal rate — a serious business problem that justifies building a predictive model.
Purpose of Day 2
This is called Exploratory Data Analysis (EDA). The rule in data science is: never touch data you have not looked at. Before cleaning, modelling, or visualising anything, you need to understand what you are working with. EDA answers the fundamental questions before you write a single analytical line of code.
Goal achieved
Established the key business metric — a 26.5% churn rate that exceeds industry norms. This single number justifies the entire project. Also flagged TotalCharges stored as text — carried into Day 3 as the first cleaning task.
Alternative approaches
df.info() shows dtypes and missing value counts in one call — useful for a quick overview. df.describe() gives statistical summary of all numeric columns. We use both in later notebooks when preparing features for machine learning.

Cleaning Summary

Fixed 3 data quality issues. Investigated 11 missing values — all brand new customers, not bad data. Final dataset: 7,043 rows x 20 columns, zero missing values. Ready for analysis.

pd.to_numeric().fillna().map().drop()Missing Value Analysis
"Notebook 2: Data cleaning — fixed TotalCharges, encoded Churn, removed CustomerID"

Every code cell explained

Fix 1 — Convert TotalCharges to a number
df['TotalCharges'] = pd.to_numeric(df['TotalCharges'], errors='coerce') print("New type:", df['TotalCharges'].dtype) print("Missing values created:", df['TotalCharges'].isnull().sum())
What this does: pd.to_numeric() converts a column to numeric format. errors='coerce' means: if any value cannot convert, turn it into NaN instead of crashing.

Result: dtype changed from object to float64. 11 values became NaN — which we then investigated.

Why not .astype(float)? That crashes immediately on any non-numeric character. pd.to_numeric(errors='coerce') handles messy real-world data gracefully without stopping your entire analysis.
Investigation — why are 11 values missing?
missing = df[df['TotalCharges'].isnull()] print(missing['tenure'].value_counts())
What this does: Filters the DataFrame to only rows where TotalCharges is null, then checks what these 11 rows have in common.

What we found: All 11 had tenure = 0. Brand new customers who have not been billed yet. Their TotalCharges being blank is expected behaviour, not an error.

Why this investigation step is critical: Without investigating, you risk making the wrong fix. In a job interview, explaining that you investigated before fixing is exactly what impresses hiring managers. It shows analytical thinking, not mechanical coding.
Fix 2 — Fill missing TotalCharges with 0
df['TotalCharges'] = df['TotalCharges'].fillna(0) print("Missing after fix:", df['TotalCharges'].isnull().sum())
What this does: Replaces every NaN in TotalCharges with 0. .fillna(value) is the standard pandas method for filling missing values.

Why 0 and not the mean? These customers genuinely have zero charges — they have not been billed yet. Filling with the mean (approximately 1,400) would be inventing false data. We fill with 0 because that is the truth. Always justify your fill value with domain knowledge, not just statistics.
Fix 3 — Encode Churn as 0 and 1
df['Churn'] = df['Churn'].map({'Yes': 1, 'No': 0}) print(df['Churn'].value_counts())
What this does: Replaces text values "Yes" and "No" with numbers 1 and 0. .map() takes a dictionary and substitutes each value according to your mapping.

Why ML needs numbers: Algorithms perform mathematical calculations. They cannot compute the average of Yes and No but they can compute the average of 1 and 0. This is called binary encoding and is standard practice for any yes/no column.

Convention: Always encode the positive event as 1. Churned = 1, stayed = 0. This is universal convention in classification problems.
Fix 4 — Drop CustomerID and save
df = df.drop('customerID', axis=1) df.to_csv('../data/churn_cleaned.csv', index=False)
What this does: .drop() removes a column. axis=1 means drop a column (axis=0 drops a row). Then we save cleaned data to a new file. index=False prevents pandas adding a row number column.

Why drop CustomerID? Every row has a unique ID — no pattern can be learned from it. Including it in a model adds noise without signal.

Why save to a new file? Never overwrite raw data. If you make a mistake in cleaning you need to go back. This principle is called data provenance — always knowing where your data came from.
Purpose of Day 3
Data cleaning is the most underestimated skill in data analysis. In real analyst jobs 60-80% of time is spent cleaning data. The goal is not to fix problems mechanically — it is to understand why each problem exists, make a justified decision, and document it so others can understand what was done and why. This is professional-level analytical thinking.
Goal achieved
Transformed a messy raw dataset with 3 quality issues into a clean analysis-ready file. Every decision was investigated and justified. Clean data saved separately from raw — the professional standard. Zero missing values. All columns in correct types. Ready for visualisation and machine learning.
Alternative approaches
For encoding Churn you could also use LabelEncoder from scikit-learn or pd.get_dummies(). For a simple binary column .map() is the most readable and explicit choice. pd.get_dummies() becomes more useful for columns with 3 or more categories — like Contract Type with its three values. We use that approach in the machine learning notebook on Day 6.

Chart 1 — Churn distribution
import matplotlib.pyplot as plt import seaborn as sns df = pd.read_csv('../data/churn_cleaned.csv') sns.set_style("whitegrid") fig, ax = plt.subplots(figsize=(8, 5)) df['Churn'].value_counts().plot(kind='bar', color=['steelblue', 'salmon'], ax=ax) ax.set_title('Customer Churn Distribution', fontsize=14, fontweight='bold') ax.set_xticklabels(['Stayed', 'Churned'], rotation=0) plt.tight_layout() plt.savefig('../reports/churn_distribution.png', dpi=150) plt.show()
New concepts introduced:

plt.subplots(figsize=(8,5)) — creates a figure of set dimensions. fig is the whole canvas, ax is the chart area inside it. Separating these gives precise control over every element.

kind='bar' — tells pandas to draw a bar chart from the value counts.

dpi=150 — dots per inch. Higher means sharper image. 150 is sufficient for portfolio screenshots. Use 300 for print quality.
Chart 2 — Churn by contract type
churn_by_contract = df.groupby('Contract')['Churn'].mean() fig, ax = plt.subplots(figsize=(8, 5)) churn_by_contract.sort_values().plot(kind='bar', color='steelblue', ax=ax) ax.set_title('Churn Rate by Contract Type', fontsize=14, fontweight='bold') ax.set_ylabel('Churn Rate') ax.set_xticklabels(ax.get_xticklabels(), rotation=0) plt.tight_layout() plt.savefig('../reports/churn_by_contract.png', dpi=150) plt.show()
New concept: groupby + mean

df.groupby('Contract')['Churn'].mean() is one of the most powerful pandas patterns. It groups all rows by Contract type, then calculates the mean Churn value for each group.

Because Churn is 0 or 1, the mean IS the churn rate. If 43 out of 100 month-to-month customers churned, the mean is 0.43 — a 43% churn rate. This pattern works for any binary outcome column and you will use it constantly as an analyst.
